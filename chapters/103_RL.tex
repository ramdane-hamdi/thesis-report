\section{Deep Reinforcement Learning}
\label{sec:reinforcement_learning}

\subsection{How an Agent Learns from Trial and Error}

\textbf{Deep Reinforcement Learning} (Deep RL) is a way of teaching a computer to make good decisions by letting it try things and learn from the results. It is different from other types of machine learning because the agent is not given the right answer. Instead, it gets a \textbf{reward} when it does something good and a \textbf{penalty} when it does something bad. Over time, it figures out which actions lead to the best outcomes \cite{sutton2018reinforcement}.

The basic idea is surprisingly simple. Imagine you are learning to ride a bike. Nobody gives you a manual with every move written down. You just try, fall, adjust, and try again. After a while, you learn to balance without even thinking about it. Deep RL works in a similar way, except the ``bike'' is a simulated environment and the ``rider'' is a software agent.

What makes Deep RL really powerful is that it can handle problems where the best action depends on the situation. There is no single correct move. The right choice at one moment might be the wrong choice a second later. This makes Deep RL a natural fit for tasks like endoscopy, where the scene keeps changing and the agent has to adapt in real time.

Some of the most famous examples of Deep RL include \textbf{DQN}, which learned to play Atari games at a superhuman level just by watching the screen \cite{mnih2015dqn}, and \textbf{AlphaGo}, which beat the world champion at Go \cite{silver2016alphago}. These results showed that Deep RL can solve problems that were considered way too hard just a few years before.


\subsection{States, Actions, Rewards and the MDP}

To train an Deep RL agent properly, we need to describe the problem in a clear and formal way. The standard framework for this is called a \textbf{Markov Decision Process} (MDP). An MDP has five parts:

\begin{itemize}
\item \textbf{State ($s$):} what the agent can observe right now. In our case, this is the image from the endoscope camera plus any extra information like tool position.
\item \textbf{Action ($a$):} what the agent can do. For navigation, this could be moving the tip up, down, left or right. For extraction, it could be extending or closing the snare.
\item \textbf{Reward ($r$):} a number that tells the agent how good the last action was. A positive reward means things are going well, a negative reward means something went wrong.
\item \textbf{Transition:} the rules that decide what happens next. After the agent takes an action, the environment moves to a new state.
\item \textbf{Discount factor ($\gamma$):} a number between 0 and 1 that controls how much the agent cares about future rewards versus immediate ones.
\end{itemize}

The ``Markov'' part means that the next state depends only on the current state and the current action, not on the entire history. This is a simplification, but it works well in practice for many problems. When we add an LSTM to the agent later, we give it a short memory to handle cases where context from a few steps back is important \cite{sutton2018reinforcement}.

The agent's goal is to find a \textbf{policy} ($\pi$), which is basically a rule that says: given this state, do this action. A good policy is one that leads to high \textbf{total reward} over time, not just a high reward on one single step. The total future reward from a state is called the \textbf{return}, and it is usually written as:

$$G_t = r_t + \gamma r_{t+1} + \gamma^2 r_{t+2} + \dots$$


\subsection{Value Methods vs Policy Methods}

There are two main ways to solve an Deep RL problem, and it is useful to understand the difference.

The first approach is called \textbf{value-based learning}. Here, the agent learns a \textbf{value function} that estimates how good each state or action is. The most famous example is \textbf{Q-learning}, where the agent learns a table (or a neural network) that scores every possible action in every possible state. Then it just picks the action with the highest score. This works great when the number of possible actions is small, like in simple grid worlds or Atari games \cite{mnih2015dqn}.

\centering

\includegraphics[width=1\linewidth]{assets/diagram-q-table.png}%
\caption{Q-Learning and the Q-table.}
\label{fig:qlearning}

The second approach is called \textbf{policy-based learning}. Here, the agent learns the action rule directly, without bothering to estimate values first. The policy is usually a neural network that takes a state as input and outputs probabilities for each action. This approach is often better for \textbf{continuous control}, where the agent needs to output smooth movements instead of choosing from a short list.

In practice, the best results often come from combining both ideas. This is called the \textbf{actor-critic} approach: the ``actor'' is the policy that chooses actions, and the ``critic'' is a value function that judges how good those actions are. The critic helps reduce the noise in the learning signal, which makes training more stable. PPO, the algorithm we use in this project, is an actor-critic method \cite{schulman2017ppo}.

\subsection{Reward Design: Why Getting the Reward Wrong Can Be Dangerous}

The reward function is probably the most important part of any Deep RL system, and also the trickiest to get right. If the reward is designed well, the agent will learn useful and safe behaviour. If the reward is designed badly, the agent can learn very strange or even dangerous things.

This problem is sometimes called \textbf{reward hacking}. The agent finds a way to get a high score without actually doing what you wanted. For example, if you reward an endoscopy agent only for ``reaching the polyp quickly,'' it might learn to crash through the colon wall because that is the shortest path. Obviously, that would be terrible in a real procedure.

That is why in medical applications, reward design has to be very careful. Some useful strategies include:

\begin{itemize}
\item \textbf{Reward shaping:} giving small intermediate rewards along the way, not just at the end. For example, a small reward for keeping the polyp in view, another for reducing distance to the target.
\item \textbf{Safety penalties:} subtracting reward when the agent does something dangerous, like touching the colon wall too hard or moving the snare too fast.
\item \textbf{Curriculum learning:} starting with an easy version of the task and making it harder over time, so the agent does not get overwhelmed early on.
\item \textbf{Simulation first:} always training in a virtual environment before even thinking about real hardware. This way the agent can make mistakes safely \cite{sutton2018reinforcement}.
\end{itemize}


In our project, the reward function is designed to balance two things: \textbf{progress} (getting closer to the goal) and \textbf{safety} (not causing damage). We will describe the exact reward design in the Contributions chapter.

\subsection{Deep RL in Medical Robotics and Endoscopy}

Deep RL is starting to be used in medical robotics, and the results so far are very promising. The main idea is always the same: train the agent in \textbf{simulation} first, then carefully test it on real hardware. This is called \textbf{sim-to-real transfer}, and it is one of the biggest challenges in the field.

Recent work by Pore et al. showed that an Deep RL agent can learn to navigate a virtual colon using only camera images as input \cite{pore2022iros}. Another study by Corsi et al. added safety constraints to the Deep RL training process, making sure the agent avoids dangerous actions even during exploration \cite{corsi2023arxiv}. In surgical robotics more generally, Ou and Tavakoli demonstrated that Deep RL agents trained in simulation can successfully transfer to a real da Vinci robot for tissue manipulation tasks \cite{ou2023lra}.

For our project, we use \textbf{Unity} as the simulation platform \cite{juliani2018unity} and \textbf{PPO} as the learning algorithm. The environment is designed to look and behave like a simplified colon, with a target polyp that the agent must reach and extract. We will describe this setup in detail in the Contributions chapter.
